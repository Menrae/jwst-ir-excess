fix: four real bugs from a 75-field smoke batch; add starved-fit gate

75-field random-sample smoke batch (pipeline/retriever.py, config/pipeline_config.yaml):
- Random draw (not stratified like the 15-field trial), 75 targets via
  random.Random(20260723).sample() over the 676-target pool (691
  archive-wide minus the 15 already tested) -- deliberately unbiased,
  since this run's purpose (bulk-query volume, real dense-field
  frequency) needed representativeness, not category coverage
- Surfaced a high-severity sampling-bias bug: pivot_to_one_row_per_star's
  .unstack("filter") call had an undocumented side effect (always sorts
  its result ascending by star_id), which silently pushed every
  Gaia-matched star behind every unmatched-sentinel star for every
  field, since sentinels are small negative integers and real
  gaia_source_ids are large positive ones. Any downstream first-N cap
  (used by both trial/smoke batch scripts) therefore excluded nearly
  all Gaia-matched -- and therefore all dual-band-capable -- stars
  whenever a field had more unmatched detections than the cap (the
  common case: 71/74 fields in this batch had zero matched stars
  survive their cap). Retroactively confirmed this also affected 11/15
  of the earlier 15-field trial's fields
- Fixed at the source: row order is now explicit first-appearance
  order, captured before unstack() and restored via reindex()
  afterward. Verified on HD-152249: capped sample went from 0->5
  Gaia-matched/dual-band stars with no other change
- Fixed the batch-script capping strategy too: new
  cap_stars_for_compute_budget(ds, max_stars, seed) (pipeline/
  retriever.py, not just the throwaway scripts) draws a random sample
  instead of a first-N prefix -- chosen over a Gaia-matched-priority
  cap because it preserves the capped sample's value as an unbiased
  population estimate rather than a deliberately non-representative one,
  with no change to the cap's worst-case compute-cost bound
- Fixed a real crash: coronagraphic (Lyot/4QPM) MIRI subarrays never
  produce a _cat.ecsv catalog (confirmed live: 24/1179 observations, 3
  targets), which used to crash load_miri_catalog_sources with
  astropy's opaque "no values provided to stack" once every observation
  for a target hit this case. Fixed at the query stage
  (query_miri_observations now excludes masklyot/mask1065/mask1140/
  mask1550), plus a defensive, clearly-worded ValueError as a backstop
  for any other future zero-catalog cause
- Fixed a real inefficiency: crossmatch_gaia launched one Gaia cone
  search per raw obs_id, so repeatedly-observed calibration targets
  triggered dozens of near-identical queries (BD+60-1753: 79
  observations, 439s of Gaia crossmatch alone). Now groups observations
  by field center (_group_observations_by_position,
  GAIA_QUERY_DEDUP_RADIUS_ARCSEC=5.0) before querying -- verified 7.3x
  real speedup (439.3s -> 60.2s) on the same field, less than a naive
  79->1 reduction would suggest since real per-observation field
  centers vary by up to 64 arcsec due to dither/subarray placement, not
  the ~0.2 arcsec the target's own nominal position implied
- Fixed the two flagged risks with real numbers: zero TAP/VizieR query
  failures across ~222 live network calls (bulk IN-clause queries hold
  up fine at this call volume, with the stated caveat that each stays
  bounded to <=20 IDs per field, so true archive-scale query-string
  length remains untested); NGC-346-scale dense fields occurred in
  1.35% of this random draw vs. the 15-field run's deliberate 6.7%
- Fixed a separate, real MAST download hang: NGC6720 stalled ~77
  minutes before a ReadTimeout, because astroquery.mast's download path
  has no timeout at all by default (confirmed in its own source;
  Conf.timeout does not apply to file downloads). Fixed via
  _DefaultTimeoutAdapter, a small requests.HTTPAdapter subclass mounted
  on Observations' shared session (DOWNLOAD_TIMEOUT_S = (30, 120)s
  connect/read) -- the standard pattern for giving a requests.Session a
  default timeout it doesn't natively support. Re-verified NGC6720
  alone: succeeded in 81.3s: the original hang was transient MAST-side
  slowness, not a persistent problem with this field or this project's
  code
- Re-ran both the 75-field batch and the original 15-field trial with
  all fixes applied. 75-field before/after: real photosphere-fit rate
  1.5% -> 7.9%, qc_candidate_preliminary 0 -> 3, qc_single_band_candidate
  0 -> 1, wall time +34% (expected: real Gaia-matched stars now
  actually enter the analysis). 15-field before/after (same 14 fields,
  NGC6720 excluded from both sides): qc_star_disqualified and
  qc_candidate_preliminary unchanged (251/253, 0) -- that run's
  zero-candidates headline holds up under the corrected sampling

pipeline/photosphere.py, pipeline/excess.py, config/pipeline_config.yaml
(starved-fit gate, following individual vetting of the first 4
qc_candidate_preliminary hits):
- Individually vetted all 4 real candidates the smoke batch produced,
  same depth as the CONTROLFIELD star 13 walkthrough. Corrected an
  initial framing in the process: not "3 fit failures, 1 real gap" as
  first assumed from the Teff-vs-classification mismatch alone, but 2
  genuine fit-quality failures (confirmed by directly computing each
  star's real chi2(Teff) curve) and 2 well-constrained, cross-validated
  fits with no identified problem
- HD-152249 and SN2017gci: both used only Gaia G/BP/RP (3 mutually
  correlated bands, no 2MASS), a near-degenerate 2-free-parameter fit
  against barely one real color constraint -- confirmed directly
  (SN2017gci's chi2 minimum was 0.136, essentially a guaranteed "good"
  fit regardless of the true Teff). Also confirmed HD-152249's own
  detection (G=19.6) cannot be the real HD 152249 (a bright ~V=5.9
  star) -- MAST's target_classification describes the field, not this
  star
- New count_effective_bands()/qc_starved_photosphere_fit
  (photosphere.min_effective_bands: 3): Gaia's G/BP/RP count as at most
  1 combined effective band; each 2MASS band counts individually.
  Wired into excess.py's DISQUALIFYING_STAR_FLAGS. A genuinely separate
  check from qc_poor_photosphere_fit -- catches "technically fittable
  but not meaningfully constrained," which a chi2 threshold alone
  cannot
- Verified three ways: 5 new pure/network-free tests
  (count_effective_bands on synthetic inputs); live re-fit of both real
  stars confirms n_effective_bands=1, qc_starved_photosphere_fit=1,
  Teff unchanged; full excess.py run confirms qc_star_disqualified=1
  with disqualifying_flags='qc_starved_photosphere_fit' as the sole
  reason for both
- Aggregate re-run impact computed retroactively against real cached
  data (no re-fitting needed): 14/1425 stars newly disqualified in the
  75-field batch (12 beyond the 2 that prompted it), 1/253 in the
  15-field batch -- confirms this gate has real reach beyond the two
  candidates that surfaced it
- For the other 2 candidates (AX-J1600.9-5142, J1757132: both 6/6
  bands, sharp Kurucz+PHOENIX-cross-validated chi2 minima, no
  crowding/saturation/neighbor found via pixel-level cutout checks),
  tested the fixed log_g=4.5 dwarf assumption as a possible explanation
  by refitting at log_g down to 2.0. Does not cleanly resolve either:
  J1757132's excess attenuates to ~0% at giant log_g (2.5) but only at
  an 18.3x worse fit to the star's own optical/near-IR colors -- a real
  tension, not a resolution; AX-J1600.9-5142's excess only drops from
  21.7% to 5.0% at best, at 3.4x worse chi2. Both remain genuinely
  unresolved and are documented individually in RESEARCH_CONTEXT.md,
  not folded into a general caveat -- J1757132 (dual-band, tightest
  chi2 minimum of the four) flagged as the most interesting result this
  project has produced so far
